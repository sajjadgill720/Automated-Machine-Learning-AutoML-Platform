1. Executive Summary

This project presents a scalable, modular, and industry-aligned Automated Machine Learning (AutoML) system designed to efficiently handle large-scale datasets containing millions of rows and thousands of features. The system automates the full machine learning lifecycle—from data ingestion to model selection and hyperparameter tuning—while explicitly incorporating intelligent data sampling to reduce computational cost without compromising decision quality.

Unlike naïve AutoML pipelines that exhaustively train models on full datasets, this system follows a progressive evaluation strategy inspired by Google AutoML / Vertex AI best practices, where a statistically representative subset of data is first used to determine the best model family. This approach enables rapid experimentation, cost efficiency, reproducibility, and scalability.

2. Problem Statement

Modern organizations generate massive volumes of data, yet most AutoML systems:

  • Assume full-dataset training
  • Are opaque and difficult to audit
  • Do not scale efficiently on local or constrained infrastructure
  • Offer limited control over preprocessing and model selection logic

Training multiple candidate models on millions of rows is computationally expensive, time-consuming, and unnecessary during early-stage model selection.

Core Problem

How can we automatically determine the most suitable machine learning model for very large datasets while minimizing computation, maintaining statistical validity, and ensuring production-readiness?

3. Proposed Solution

This project introduces a sampling-first AutoML architecture that:

  1. Accepts large datasets (CSV, DataFrame, API input)
  2. Applies intelligent sampling (e.g., 5,000 representative rows)
  3. Runs a full AutoML pipeline on the sampled data
  4. Identifies the best model family and configuration
  5. Optionally escalates to full-dataset training for final deployment

This mirrors how large-scale ML systems operate in real industry environments.

5. Intelligent Sampling Strategy (Key Contribution)

5.1 Motivation

Training all models on full datasets:

  • Increases runtime exponentially
  • Consumes excessive memory and energy
  • Slows experimentation cycles

Industry research shows that model ranking stabilizes early, meaning the best model on a sample is usually the best model on the full dataset.

5.2 Sampling Design

Sampling is implemented as a first-class preprocessing step:

  • Random sampling for regression tasks
  • Stratified sampling for classification tasks
  • Configurable sample size (default: 5,000 rows)
  • Fixed random seed for reproducibility

Sampling occurs before preprocessing and feature selection, ensuring all downstream steps benefit from reduced data volume.

5.3 Alignment with Google AutoML Standards

Google AutoML systems follow a two-phase approach:

  1. Exploration Phase: Sampled data used to identify optimal model architectures
  2. Exploitation Phase: Selected models trained on larger or full datasets

This project explicitly implements this philosophy, making it compliant with large-scale ML operational standards.

6. Data Preprocessing Framework

A dispatcher pattern routes data to specialized preprocessing modules:

Supported Data Types

  • Tabular
  • Text
  • Image
  • Time Series

Key Features

  • Missing value handling
  • Encoding and scaling
  • Train/validation/test splitting
  • Leakage-safe transformations
  • Artifact tracking (scalers, encoders)

This design ensures consistency and reproducibility across runs.

7. Feature Selection Module

Feature selection is applied after preprocessing and before model training to:

  • Reduce dimensionality
  • Improve training speed
  • Prevent overfitting

Implemented Methods

  • Variance Threshold
  • Recursive Feature Elimination (RFE)
  • Model-based importance (tree models)
  • Combined robust selection strategy

Selected feature indices are applied consistently across all dataset splits.

8. Model Training Framework

Multiple baseline models are trained to avoid algorithmic bias:

Classification

  • Logistic Regression
  • Decision Tree
  • Random Forest
  • Gradient Boosting
  • Support Vector Machine
  • k-Nearest Neighbors

Regression

  • Linear Regression
  • Ridge / Lasso
  • Random Forest Regressor
  • Gradient Boosting Regressor

Each model is trained under identical conditions for fair comparison.

9. Evaluation & Model Selection

Models are evaluated using task-appropriate metrics:

Classification

  • Accuracy
  • Precision (weighted)
  • Recall (weighted)
  • F1-score (weighted)
  • Confusion matrix

Regression

  • RMSE
  • R²

The best model is selected deterministically using a clear, explainable hierarchy of metrics.

10. Hyperparameter Tuning

Only the selected best model undergoes tuning, preventing unnecessary computation.

Supported Methods

  • Grid Search
  • Random Search
  • Bayesian Search (Optuna-style)

This mirrors industry practice where tuning is applied selectively.

11. Pipeline Orchestration

The run_pipeline() function serves as the single public API:

  • Coordinates all modules
  • Enforces execution order
  • Returns JSON-safe structured results
  • Supports programmatic, CLI, and API usage

This makes the system suitable for automation, deployment, and UI integration.
